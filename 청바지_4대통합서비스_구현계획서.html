청바지(NowYouth) — 4대 통합 서비스 구현 상세 계획서

플랫폼 내 서비스 제공·구현 방식 완전 명세

문서 버전: v1.0 (2026-04-19) 범위: 청바지가 제공할 4대 핵심 서비스(복지 알리미 · 활동 리워드 · 일자리 매칭 · 커뮤니티(품앗이))가 플랫폼 내에서 어떻게 제공되고 현실적으로 구현되는지 완전 명세 독자: AC 심사위원 · 파트너 기관 · 엔지니어링 팀 · 실증 지자체 담당자

0. 통합 철학 — "4개 앱이 아니라 1개 일상"

청바지는 4가지 서비스를 나열한 앱이 아니라, 시니어의 하루 전체를 덮는 단일 레이어다. 4대 서비스는 각각 독립적 가치를 제공하면서도, 하나의 포인트·프로필·커뮤니티 엔진을 공유하여 교차 시너지를 만든다.

왜 통합인가: - 시니어는 앱을 여러 개 설치하고 전환하지 않는다. 단일 진입점이 필수. - 4대 서비스는 같은 프로필(연령·거주지·소득·관심사)을 기반으로 하므로 중복 입력을 피할 수 있다. - 한 서비스의 행동이 다른 서비스의 데이터가 된다 (예: 품앗이에서 드러난 기술 → 일자리 매칭 입력).

서비스 핵심 가치 주 데이터 소스 수익 연결 S1 범위 S3 범위
① 복지 알리미 "내가 받을 수 있는 혜택 자동 알림" 복지로 · 공공데이터포털 · 정부24 B2G · 정부 위탁 Rule 기반 매칭·수동 요약 AI 예측·자동 요약·지자체 직연동

③ 일자리 매칭 "나에게 맞는 일을 미리 찾아줌" 노인인력개발원 · 수행기관 B2G SaaS (수행기관 월 구독) 공공 API 목록·수동 전달 AI 매칭·수행기관 백오피스

4대 서비스 통합 KPI (핵심 1개): "월간 활성 사용자(MAU) 중 2개 이상 서비스를 사용한 비율" → 목표 S1 30% · S2 50% · S3 70%.

2. ① 복지 알리미 — 구현 상세

2.1 서비스 정의

"시니어가 받을 수 있지만 모르고 지나치는 복지 혜택을 자동으로 찾아 알려주고, 신청 완료까지 도달하도록 끌고 가는 서비스."

핵심 문제: 기초연금 사각지대 42만 명, 문화누리카드 70세+ 이용자 8.2% 미소진 — "알림이 닿지 않아서" 발생하는 사각지대.

2.2 사용자 경험 흐름 (구체 시나리오)

시나리오 A — 아침 푸시 → 신청 완료

08:00 푸시 알림
  "어머니, 오늘 받을 수 있는 혜택 1건이 있어요.
   [기초연금 소득역전 감액 완화 적용] 월 약 4만원 추가"
        ↓
사용자 앱 진입 → 카드 상세 화면
  · 내가 받을 수 있는 이유 (요약 3줄)
  · 필요 서류 체크리스트 (주민등록등본·소득 증빙)
  · 가까운 행정복지센터 (Kakao Map 버튼)
  · 예상 소요 시간 "주민센터 방문 30분"
        ↓
[신청하러 가기] 버튼
  → 신청 상태 "대기 중"으로 표시
  → 완료 시 [완료 체크]로 변경 → 100 포인트 적립

시나리오 B — 챗봇 질의

사용자 음성 입력: "나 이번달에 뭐 받을 수 있어?"
        ↓
AI 챗봇 응답 (TTS 낭독):
  "어머님, 이번 달 받으실 수 있는 혜택 3가지가 있어요.
   첫째, 에너지바우처 겨울 지원 약 11만원.
   둘째, 이동지원 서비스 월 8회 무료.
   셋째, 어르신 일자리 신청 기간이 다음 주 시작돼요."
        ↓
"첫번째 자세히 알려줘" → 카드 전환

시나리오 C — 신청 실패 구제

복지로 API에서 신청 결과 "반려" 감지
        ↓
푸시: "반려된 신청이 있어요. 서류 보완 도움 받으세요"
        ↓
옵션 제시:
  · 필요 서류 재안내
  · 동네 품앗이 커뮤니티에서 "주민센터 동행" 요청글 작성 ← ④ 연결
  · 관할 사회복지사 연결 (S3)

2.3 데이터 소스·수집 파이프라인

소스 제공 내용 수집 방식 갱신 주기
복지로 Open API 중앙정부 복지 제도 약 360종 + 자격 요건 일 1회 Cron 전체 동기화 매일 새벽 4시
공공데이터포털 지자체 복지사업 DB · 행정구역 코드 월 1회 배치 월초
보건복지부 보도자료 RSS 신규 시행·개편 소식 RSS 폴링 30분 실시간
지자체 홈페이지 크롤링 (S2+) 지자체별 로컬 혜택 주 1회 크롤링 주간
정부24 알림 서비스 API (S3) 사용자별 개인화 수급 가능 제도 기관 협약 후 실시간 실시간

2.4 매칭·추천 로직 (단계별 진화)

S1 — Rule Engine (SQL)

-- 자격 조건 필터
SELECT * FROM welfare_programs
WHERE
  :user_age BETWEEN age_min AND age_max
  AND :user_dong_code = ANY(region_codes)
  AND (:user_income IS NULL OR :user_income <= income_max)
  AND household_type @> :user_household
  AND deadline > NOW()
ORDER BY benefit_amount_krw DESC
LIMIT 10;

S2 — Rule + AI Ranker (pgvector)

Candidate (SQL) → Top 50
        ↓
[pgvector 유사도] 사용자 행동 벡터 × 혜택 임베딩
        ↓
[LightGBM] 수급 전환율 예측 (과거 클릭·신청·완료 로그 학습)
        ↓
[LLM Re-rank] 시니어 친화 설명문 생성 · 상위 5~7건

S3 — 예측 모델

사용자 소득·가구 변화(국민연금 수급 시작, 배우자 사망 등) 감지 시 선제적 알림. 변경 이벤트는 정부24 API 또는 사용자 직접 입력으로 확보.

2.5 콘텐츠·편집 운영 (숨은 핵심)

원문 그대로는 시니어가 읽을 수 없다. 예시:

  • 원문 (복지로): "65세 이상 소득인정액 기준 선정기준액 이하 가구 구성원에게 지급하는 공적이전소득으로서..."
  • 편집본 (청바지): "만 65세 넘고, 가족 포함 월 소득이 187만원 이하면 받을 수 있어요. 매달 통장으로 최대 34만원."

편집 운영 실무:

단계 편집 주체 월간 발행량 품질 관리
S1 주상윤·김경림 수동 작성 주 2회 · 월 8건 베타 사용자 피드백
S2 전담 콘텐츠 에디터 1명 + LLM 초안 월 20건 시니어 리서처 주 1회 감수
S3 에디터 2명 + AI 자동 초안 월 100건 (지역별 맞춤) 접근성·언어 품질 CI 검사

2.6 다른 서비스와의 통합

  • ② 리워드: 혜택 신청 완료 시 100 포인트 적립. 반려 보완 제출 시 추가 50P.
  • ④ 커뮤니티: 신청 어려움 겪는 사용자가 품앗이 "주민센터 동행" 요청글 1-탭 작성.
  • ③ 일자리: 복지 혜택 중 "노인일자리 사업"은 일자리 서비스와 통합 카드로 노출.

3. ② 활동 리워드 시스템 — 구현 상세

3.1 서비스 정의

"일상의 작은 활동(게임·걷기·문화·학습·이벤트)이 포인트로 쌓이고, 원하는 것으로 교환되는 경험."

이 서비스는 독립적 가치(건강·재미)도 있지만, 4대 서비스 전체의 리텐션 엔진이다. 혜택 신청·일자리 참여·품앗이 도움도 전부 포인트로 연결된다.

3.2 활동 카탈로그 (포인트 적립 수단 전체)

카테고리 활동 적립 단위 구현 단계 증빙 방식
인지 훈련 숫자 기억·한글 맞추기·초성 퀴즈 게임 1판 완료 = 20P S1 앱 내 이벤트
걷기 일일 걸음 수 1,000보 = 10P (일 최대 100P) S2 HealthKit/Google Fit (네이티브 앱)
지하철 탐방 서울 지하철 역 QR 스탬프 1역 스탬프 = 50P S2~S3 GPS + QR 스캔 (지자체·코레일 협업)
문화 활동 공연·전시·박물관 방문 1회 인증 = 200P S2 문화누리카드 결제 Webhook(S3) or 티켓 사진 업로드(S2)
학습 혜택 학습 퀴즈 (오늘의 혜택 내용 맞추기) 1문제 정답 = 10P S1 앱 내 이벤트
이벤트 지자체 오프라인 스탬프 투어 이벤트별 상이 S2 기관 담당자 QR 발급
혜택 신청 복지 혜택 신청 완료 1건 = 100P S1 외부 시스템 확인 or 사용자 인증샷
일자리 참여 노인일자리 신청·근무 완료 1회 = 200P S2 수행기관 확인
품앗이 이웃 도움 제공 / 수령 1회 = 100P 양쪽 S1 매칭 완료 + 상호 평가

3.3 포인트 원장 (Ledger) 설계

불변 append-only 원칙 — 감사·정산·분쟁 대응에 필수.

point_ledger (
  id UUID PK,
  user_id UUID,
  type ENUM(WALK, GAME, STAMP, CULTURE, LEARN, EVENT, WELFARE, JOB, POOMASI, REDEEM, ADMIN_ADJUST),
  amount INT,
  metadata JSONB,  -- 예: {game_id, session_id, steps, station_code}
  source ENUM(USER_ACTION, SYSTEM_AWARD, ADMIN_CORRECTION),
  created_at TIMESTAMPTZ,
  immutable BOOLEAN DEFAULT TRUE
)

잔액 계산: 매회 집계 대신 user_balance 뷰 또는 자동 갱신 테이블 유지 — 대규모 내역이 쌓여도 조회 빠름.

3.4 리워드 카탈로그·교환

단계 교환 옵션 처리 방식
S1 편의점 상품권 1,000P = 1,000원 월 1회 수기 발송 (우리 팀 직접 구매·우편)
S2 카카오톡·네이버페이 기프티콘 30종 API 자동 발급 (파트너십 계약 후)
S2 약국·문화시설 쿠폰 부분 파트너십
S3 지자체 특산품 (지자체별 상이) B2G 계약 옵션
S3 문화누리카드 바우처 충전 카드사 제휴 전제

원칙: 현금성 환급 금지(금융 규제) · 건강·문화·학습·식품 중심.

3.5 이벤트 운영 모델

지하철 스탬프 투어 (핵심 킬러 이벤트)

  • 시니어 지하철 무료 탑승 혜택과 결합 — 이동 비용 0원
  • 월 1회 테마 투어: "서울 역사 문화 5역 스탬프" 또는 "공원 5곳 스탬프"
  • 지자체 또는 서울교통공사와 협업 → 주요 역사에 QR 스티커 배치
  • 완주 시 특별 배지 · 500P 보너스 · 소정의 경품

구현 단계: - S1: 미지원 (S2 이후) - S2: 자체 스티커 주요 5개 역 시범 (사전 협조 공문만 제출) - S3: 코레일·지자체 공식 협약, 역사 내 배너·시니어센터 가이드 배포

경로당·노인복지관 오프라인 이벤트

  • 수행기관 담당자가 QR 코드 발급 → 참여 시니어 스캔으로 포인트 지급
  • 프로그램 참여 유도에 활용 (치매 예방 교실, 건강 체조 등)

3.6 다른 서비스와의 통합

  • ① 복지: 혜택 신청 완료 · 학습 퀴즈 → 포인트
  • ③ 일자리: 신청·근무 완료 → 포인트
  • ④ 품앗이: 도움 주고받음 → 양쪽 포인트
  • 리워드 역방향: 포인트로 문화누리카드 충전(S3) · 교통카드 충전(S3)

3.7 성공 지표

단계 핵심 지표
S1 사용자당 월 평균 적립 포인트 · 교환 신청률
S2 일일 활성 사용자(DAU) · 활동 다양성 지수(한 주에 사용한 활동 종류 수)

4. ③ 노인일자리 매칭 — 구현 상세

4.1 서비스 정의

"공공 노인일자리(연 115만 개)를 시니어와 수행기관 양쪽에 직접 연결하고, 장기적으로 수행기관 행정 자동화 SaaS로 확장."

시장조사 근거(청바지_시장조사_노인일자리.md): - 수요 충족률 46.4%, 대기자 9만 1,245명 (전년 +62%) - 담당자 1인당 280명 관리 과부하 · 41.5%가 "행정 과다" 호소 - 2026.4.16 한국노인인력개발원장 AI 디지털 전환 공식 선언 - 공공 노인일자리 + 수행기관 SaaS = 경쟁자 부재 블루오션

5. ④ 커뮤니티(품앗이) — 구현 상세

5.1 서비스 정의

"행정동 단위 소규모 커뮤니티에서 이웃이 서로 도와주는 상호부조 네트워크. 금전 거래 없음, 포인트로만 교환."

핵심 문제: 시니어 고립 — 단순한 혜택 제공만으로는 해결 불가. "연결"이 실제 가치.

5.2 사용자 경험 흐름

요청자 측

  → [요청글 작성] 버튼

  → 제목·내용·희망 시간 입력 (음성 입력 가능 — Whisper S2)
  → 게시 → 같은 동 이웃에게 푸시
  → 수락자 프로필 확인 → [수락] → 실명·전화번호 공개

도움 제공자 측

  → 관심 요청글 [지원하기]

  → 완료 후 서로 별점·후기 → 100P 양쪽 적립

  2. 운영진 응답 보장 — 초기 요청글에 최소 1명 이상 수락자 확보 (우리 팀 또는 계약된 자원봉사자)
  3. 지자체 사회복지사 안내 — 경로당 단체 가입 권유

6. 4대 서비스 교차 시너지 — 통합 플랫폼의 진짜 가치

4대 서비스가 각각 독립적으로 존재하는 것이 아니라, 서로를 강화하는 구조를 명시한다.

시너지 매트릭스

구분 ① 복지 ② 리워드 ③ 일자리 ④ 품앗이
① 복지 — 신청 완료 → 100P 혜택 중 일자리 사업 카드 신청 도움 요청글 원터치
② 리워드 포인트로 문화 바우처(S3) — 신청·근무 → 포인트 도움 → 양쪽 포인트
③ 일자리 일자리 관련 지원금 혜택 교차 노출 포인트 전환 수당 — 품앗이 기술 → 일자리 태그
④ 품앗이 혜택 신청 병목 해소 포인트 적립원 비공식 소일거리 카테고리 —

핵심 시너지 3가지 (AC 심사 답변용)

시너지 1 — "단일 프로필로 4배 가치" - 온보딩에서 입력한 4개 필드(생년월일·행정동·가구·관심사)가 4대 서비스 모두에 활용. 경쟁사는 각각 따로 입력 필요.

시너지 2 — "리워드 엔진이 모든 서비스의 재방문을 유도" - 혜택 신청만으로는 월 1~2회 접속에 그침. 리워드·품앗이가 일 단위 접속을 만들어 리텐션 확보 → 그래서 혜택도 놓치지 않음.

시너지 3 — "데이터 교차 학습" - 품앗이에서 "컴퓨터 가르쳐드립니다" 글을 자주 쓴 사용자 → 일자리 매칭에 IT 강사 카테고리 자동 추가 - 복지 혜택 신청 포기율 높은 지역 → 품앗이 "신청 동행" 요청을 우선 노출 - 이런 학습은 통합 플랫폼에서만 가능한 경쟁 우위.

10. 통합 KPI 대시보드 설계

서비스별 지표와 통합 지표를 분리 운영

서비스별 지표 (각 서비스의 건강성)

  • ① 복지: 혜택 노출 수 · 신청 전환율 · 수급 완료율
  • ② 리워드: 적립 포인트 · 교환 신청률 · 활동 다양성
  • ③ 일자리: 매칭 성사율 · 수행기관 만족도 · ARR
  • ④ 품앗이: 매칭 성사율 · 신고율 · 재참여율

통합 지표 (통합 플랫폼 차별화)

  • 크로스 사용률: MAU 중 2개 이상 서비스 사용 비율 → S1 30% · S2 50% · S3 70%
  • 플랫폼 가치 점수: 사용자당 월 적립 포인트 평균 · 인당 연간 수급 혜택 총액
  • 리텐션 (D30): 가입 30일 후 재접속 비율 → S1 40% · S2 55% · S3 70%
  • 고립 지표 (S3): 주 1회 이상 접속 + 품앗이 또는 커뮤니티 상호작용 있는 사용자 비율

지표 수집·리뷰 주기

  • 매일: 푸시 오픈율 · 홈 피드 클릭률 · 주요 서비스 진입률
  • 주간: 사용자 1:1 인터뷰 3건 (주상윤·김경림 공동)
  • 월간: 4대 서비스별·통합 KPI 대시보드 리뷰 미팅
  • 분기: 전략 재조정 — 기능별 리소스 재배분 여부 결정
