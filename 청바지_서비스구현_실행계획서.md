# 청바지(NowYouth) — 서비스 구현·전달 실행 계획서

**AC 투자 기획서 보완 문서 — 기술 구현 전략 + 서비스 전달 로드맵**

> **문서 버전**: v1.0 (2026-04-19)
> **용도**: AC 심사위원·파트너·합류 예정 팀원에게 "청바지가 실제로 어떻게 만들어지고, 사용자에게 어떻게 전달되는지" 명확히 설명
> **포지션**: AC 기획서(비전·시장·BM)와 MVP 개발계획서(12주 스프린트) 사이의 **3년 서비스 진화 계획**

청바지는 3개의 계층적 문서로 관리된다. 본 문서는 그 중간 레이어 — **"현실적으로 어떻게 구현하고 무엇을 전달할지"** 를 다룬다.

| 문서 | 시점 | 독자 | 상세도 |
|-----|-----|-----|-------|
| `청바지_AC투자_기획서.md` | 0~3년 | AC·투자자 | 비전·시장·BM·아키텍처 개요 |
| **`청바지_서비스구현_실행계획서.md` (본 문서)** | **0~36개월** | **투자자·파트너·팀원** | **단계별 구현·전달 전략** |
| `청바지_MVP_개발계획서.md` | 0~12주 | FE/BE 엔지니어 | 스프린트 단위 실행 |

## 2. 서비스 구현 4단계 진화 모델

청바지는 단일 시점의 제품 출시가 아니라 **4단계로 진화하는 서비스**다. 각 단계는 명확한 산출물·사용자 규모·수익·팀 규모를 가진다.

| 단계 | 기간 | 핵심 산출물 | 사용자 | 수행기관 | 팀 규모 | 투자 단계 |
|------|------|-------------|--------|----------|---------|----------|
| **S0 · Paper/Prototype** | 0~3M | 기획서·목업·PoC PWA | 내부 10명 | — | 4명 (대표·기획·FE·BE) | 자기자본 |
| **S1 · Pilot Product** | 3~6M | MVP 실가동 + 실증 1곳 | 지인 + 파일럿 50~200명 | 1곳 (무상) | 4~5명 | 시드 유치 직전 |
| **S2 · Scalable Service** | 6~18M | 유료 구독 + B2G 첫 계약 | 1,000~10,000명 | 3~10곳 (일부 유료) | 8~12명 | 시드 10억 집행 |
| **S3 · Platform/SaaS** | 18~36M | 멀티테넌트 B2G SaaS + 데이터 자산 | 30,000~100,000명 | 30~100곳 | 20~30명 | Series A 준비 |

> **핵심 원칙**: **한 단계가 완성되기 전에 다음 단계로 넘어가지 않는다.** "사용자 1,000명이 실제로 한 달 쓰는 서비스"가 없는 상태에서 멀티테넌트 SaaS를 만들지 않는다.

## 3. 각 단계에서 **실제로 전달되는 서비스** (Service Blueprint)

본 섹션은 기술 아키텍처가 아니라 **사용자가 실제로 경험하는 것**을 단계별로 서술한다.

### 📱 시니어 사용자 관점 — "매일 아침 청바지를 열면 무엇이 있는가"

| 단계 | 시니어가 실제로 경험하는 것 |
|------|----------------------------|
| **S0** | (사용 불가 — 내부 데모만) |
| **S1** | 아침 8시 푸시: "오늘 받을 수 있는 혜택 1건이 있어요" → PWA 열면 카드 1장 → 탭하면 신청 서류 리스트 + 근처 행정복지센터 주소. 품앗이 게시판에서 같은 동 이웃 글 3~5개. 포인트는 수기 발급 편의점 상품권 교환. |
| **S2** | 개인화 피드 5~7장 (혜택·일자리·활동·품앗이 혼합). AI 챗봇에게 음성으로 "나 이번달에 받을 수 있는 거 뭐 있어?" → 읽어주는 답변. 걸음 수·인지 게임으로 포인트 자동 적립 → 카카오 기프티콘 즉시 발급. 품앗이 매칭 수락 알림. |
| **S3** | 시니어가 속한 **지자체 브랜딩으로 청바지 제공** (예: "서초구 어르신 앱"). 지자체 사회복지사가 긴급상황(낙상·고립) 감지 시 직접 연락. 포인트 → 지자체 특산품 교환 추가. |

### 👪 보호자·가족 관점

| 단계 | 보호자가 실제로 할 수 있는 것 |
|------|------------------------------|
| **S1** | 대리 신청 계정으로 로그인 → 부모님 혜택 체크 대신해서 신청 |
| **S2** | 부모님 활동 요약 주 1회 리포트 (걸음 수·포인트·품앗이 참여) · 이상 패턴 경고 (며칠째 앱 미사용) |
| **S3** | 응급 연락 네트워크 등록 · 지자체 모니터링과 연동 |

### 🏢 수행기관·지자체 담당자 관점

| 단계 | 담당자가 실제로 쓰는 기능 |
|------|--------------------------|
| **S1** | 실증 파일럿 1곳 담당자가 월 1회 간단 대시보드 보고서 PDF 수령 (우리 팀이 수동 생성) |
| **S2** | 웹 대시보드 로그인 → 관할 행정동 사용자 수·품앗이 활동량·복지 수급 전환율 조회 · 엑셀 내보내기 |
| **S3** | 완전한 B2G SaaS — 담당자가 청바지 앱 사용자를 직접 관리 · 고위험군 스크리닝 · 지자체 브랜딩 설정 |

### 🛠 우리 팀 운영 관점 — "우리는 매일 무엇을 해야 하는가"

| 단계 | 매일/매주 해야 하는 일 |
|------|----------------------|
| **S1** | 50명 시니어 1:1 전화 CS (주상윤·김경림) · 매주 수행기관 미팅 · 수기 상품권 발송 |
| **S2** | CS 팀 2명 채용 · 콘텐츠 에디터 1명 (주간 복지 소식 카드) · 장애 대응 온콜 · 신규 지자체 영업 |
| **S3** | B2G 세일즈팀 3명 · 데이터 사이언티스트 · 정책 리에종 · 시니어 UX 리서처 상주 |

## 4. 기술 구현 전략 — 아키텍처가 단계별로 진화하는 방식

**핵심 원칙: 오버엔지니어링을 피하되, 전환 비용이 높은 결정은 S0에서 미리 내린다.**

### 🟢 Stage 0~1 — 단일 앱·단일 DB (심플)

```
[Next.js PWA (Vercel)]
        ↓
[NestJS 단일 서버 (Fly.io)]
        ↓
[PostgreSQL 1개 인스턴스] + [Redis] + [S3]
        ↓
[공공 API 수집 크론 잡]
```

**특징**:
- 서비스 분리 없이 모듈로만 구분
- 공공 API는 일 1회 배치 수집 → DB 캐시
- 트래픽 수백~수천 MAU 전제
- **pgvector는 미설치** (S2에서 추가)

**왜 이렇게**: 사용자 50~200명 수준에서 MSA·K8s·서비스 메시는 시간 낭비. 단일 DB 백업과 롤백이 훨씬 안전.

### 🟡 Stage 2 — 서비스 분리·큐·캐시 (확장)

```
[Next.js PWA + Native App Wrapper]
        ↓
[API Gateway (NestJS + Cloudflare Workers)]
   ↓       ↓       ↓       ↓       ↓
[복지] [포인트] [일자리] [품앗이] [알림]  ← 서비스 모듈 분리, 단일 DB 공유
        ↓
[PostgreSQL + pgvector] [Redis 클러스터] [S3]
        ↓
[BullMQ 큐] — 공공 API 수집·알림 발송·AI 요청 큐잉
        ↓
[OpenAI API Gateway + LLM 응답 캐싱]
```

**특징**:
- 서비스 모듈 간 인터페이스를 명시적으로 (메서드 호출 → HTTP/tRPC)
- pgvector 추가로 AI Ranker·유사 품앗이 추천 가능
- BullMQ로 외부 API 호출·AI 호출을 백그라운드 큐에 위임
- 장애 격리 향상

**전환 트리거**: MAU 5,000 돌파 또는 수행기관 3곳 계약 시

### 🔴 Stage 3 — 멀티테넌트·데이터 플랫폼 (B2G SaaS)

```
[PWA] [Native App] [지자체 담당자 대시보드(별도 도메인)]
           ↓
    [API Gateway + 테넌트 라우팅]
           ↓
  [복지][포인트][일자리][품앗이][알림][B2G 관리][분석]
           ↓                        ↓
[Operational DB (테넌트별 격리)] [Analytics Warehouse (ClickHouse)]
           ↓                        ↓
   [공공 + 파트너 API]         [Data Studio + 지자체 맞춤 리포트]
```

**특징**:
- 지자체별 **데이터 격리** (스키마 분리 또는 row-level policy)
- 운영 DB와 분석 DB 분리 — 지자체 대시보드가 운영에 영향 주지 않음
- 익명화 데이터 레이크 → 정책 인사이트·B2G 분석 상품화
- 멀티리전·고가용성 구성

**전환 트리거**: B2G 유료 계약 5곳 이상 + 시리즈 A 완료

## 5. 4대 기능 — **현실적으로 어떻게 만드는지** (실행 관점)

각 기능은 "기획 → 아키텍처" 순이 아니라 **"데이터 확보 → 가공 → 사용자 가치 → 지속 운영"** 4단계로 현실적으로 구축된다.

### ① 복지 알리미

| 단계 | 현실적 구현 |
|------|------------|
| **S1 데이터 확보** | 복지로 Open API 일 1회 수집 → Postgres 캐시. 공공데이터포털 행정구역 코드 매핑. |
| **S1 가공** | 보건복지부 원문 표현 → 시니어 친화 요약 (**초기에는 수동 편집**, 주상윤·김경림이 주 2회 3~5건 다듬음). |
| **S1 매칭** | SQL WHERE 절로 연령·지역·소득 구간 필터 → Top 5 카드 노출. AI 미사용. |
| **S2 자동화** | OpenAI로 요약 자동 생성 → 사람 검수만. pgvector로 "비슷한 혜택 받은 사용자들의 수급 패턴" 기반 추천. |
| **S3 고도화** | 지자체별 복지사업 DB 통합 (개별 계약). 수급 예측 모델(LightGBM)로 "당신은 이 혜택을 받을 확률 87%" 점수 제공. |
| **운영 부담** | S1: 주 6시간 수동 편집. S2: 주 2시간 검수. S3: 전담 콘텐츠 에디터 1명. |

**시니어가 받는 가치의 진화**: 
- S1: "내가 모르던 혜택 1건을 알게 됨"
- S2: "내가 받을 수 있는 혜택 3건을 미리 알려줌"
- S3: "내 소득 변화에 맞춰 실시간으로 받을 수 있는 혜택이 갱신됨"

### ② 활동 포인트

| 단계 | 현실적 구현 |
|------|------------|
| **S1 데이터 수집** | 인지 훈련 게임(자체 개발) 완료 이벤트만. 걷기·지하철·문화는 **미지원**. |
| **S1 적립·교환** | `point_ledger` 테이블. 교환은 **월 1회 수기** — 우리가 편의점 상품권 구매해서 직접 발송. 소량(월 20건) 감당 가능. |
| **S2 자동화** | 네이티브 앱 빌드 → HealthKit/Google Fit 걸음 수 자동 적립. 카카오 기프티콘 API로 즉시 발급. |
| **S2 리워드 다각화** | 편의점 + 약국 + 문화시설 기프티콘 카탈로그 30종. |
| **S3 문화누리카드 연동** | 카드사 제휴 완료 시 결제 내역 Webhook → 자동 포인트 적립. |
| **운영 부담** | S1: 월 2시간 상품권 발송. S2: 자동화 후 월 1시간 정산 확인. S3: 전담 운영 필요 없음. |

**시니어가 받는 가치의 진화**:
- S1: "게임 하면 한 달에 편의점 상품권 1장 받음"
- S2: "산책·게임·문화활동으로 매일 조금씩 쌓여 원하는 기프티콘 교환"
- S3: "생활 전체가 포인트로 연결 — 혜택 신청·품앗이·운동 전부 통합"

### ③ 일자리 매칭

| 단계 | 현실적 구현 |
|------|------------|
| **S1 공급 확보** | 한국노인인력개발원 Open API 일 1회 수집. 행정동별 공고 인덱싱. |
| **S1 매칭** | 키워드 · 거리(Haversine) · 시간대 필터. AI 없이 "가까운 일자리 N건" 노출. |
| **S1 신청** | 앱에서 "관심 있음" 체크 → 담당자 이메일로 신청서 이관 (우리가 수동). |
| **S2 AI 매칭** | 경력 태그·선호 시간대 → pgvector 코사인 유사도. 수행기관 담당자용 웹 대시보드(매칭 목록). |
| **S2 대기자 알림** | 공급 대비 수요 초과(현재 시장 46.4% 충족률)의 공백을 활용 — 새 공고 발생 시 조건 맞는 대기자에게 즉시 푸시. |
| **S3 수행기관 SaaS** | 담당자 1인당 280명 관리 → 자동화로 50명 수준 절감. 청바지가 수행기관 행정 백오피스 대체. |
| **운영 부담** | S1: 주 1~2건 신청 수동 이관. S2: 담당자 셀프서비스 전환. S3: B2G 계약 관리. |

**핵심 레버리지**: 시장조사 보고서(`청바지_시장조사_노인일자리.md`)에서 확인된 **수행기관 담당자 1인당 280명 관리 과부하·대기자 9만명 급증**이 이 기능의 근거. S3에서 B2G SaaS 수익 핵심이 됨.

### ④ 품앗이 커뮤니티

| 단계 | 현실적 구현 |
|------|------------|
| **S1 매칭 구조** | 행정동 지오펜싱(주소→좌표 Kakao Map API) · 게시판 · 1:1 요청/수락 · 휴대폰 번호 마스킹 노출. |
| **S1 안전 장치** | 본인 인증 필수 · 3회 신고 시 자동 정지 · 금융/종교 카테고리 금지 키워드 필터. |
| **S1 시드 콘텐츠** | 우리 팀이 실증 지역에서 초기 요청글 20건 시딩 (초기 사용자 외로움 방지). |
| **S2 매칭 추천** | 경력·시간대 기반 "이 요청 당신과 잘 맞아요" 추천. 양방향 평점. |
| **S3 지자체 연동** | 사회복지사가 고위험 사용자(며칠째 접속 없음) 감지 → 품앗이 요청 유도. |
| **운영 부담** | S1: 매일 신고 확인(평균 10분). S2: 커뮤니티 매니저 1명. S3: 지자체 담당자 역할 분담. |

**시니어가 받는 가치의 진화**:
- S1: "같은 동네 이웃 3명과 김장 품앗이"
- S2: "비슷한 상황의 이웃이 자동 추천됨"
- S3: "외로울 때 동네 이웃이 자동으로 연결됨 — 고립 방지 시스템"

## 6. 데이터 확보·운영 전략

### 데이터 파이프라인 (단계별)

```
S1: [공공 API] → [Cron Job (node-cron)] → [Postgres 테이블] → [앱 조회]
                    ↓ 일 1회, 실패 시 Slack 알림

S2: [공공 API + 파트너 API] → [BullMQ 큐 + 재시도] → [Postgres + Redis 캐시]
                              ↓ 실시간 · 이벤트 기반

S3: [공공 + 파트너 + 사용자 이벤트] → [Kafka] → [Operational DB + ClickHouse]
                                       ↓ 분석·AI 학습 병렬
```

### 사용자 데이터 자산화 경로

| 단계 | 축적 데이터 | 가치 |
|------|------------|------|
| **S1** | 지역·연령별 혜택 클릭·신청 전환율 | 내부 제품 개선 |
| **S2** | 복지·활동·일자리 크로스 행동 데이터 | AI 추천 학습셋 · 콘텐츠 우선순위 |
| **S3** | 수년간 수급 패턴·전환 트리거 | **정책 인사이트 B2G 라이선스 수익** · 학술 연구 협업 |

### 개인정보·규제 준수 원칙

1. **최소 수집**: 혜택 매칭에 불필요한 항목 수집 금지 (예: 혈액형·질병 상세)
2. **민감정보 분리 저장**: 건강 정보는 별도 암호화 테이블 (S2부터)
3. **3년 자동 파기 배치**: `scheduled_deletion` 테이블 + 월 1회 크론 실행
4. **고령자 보호**: 약관 동의 화면에서 **음성 낭독 기능 기본 제공** (S1부터)
5. **감사 로그**: 모든 포인트·혜택·품앗이 거래 불변 원장

## 7. 서비스 운영 체계

**청바지는 기술 제품이 아니라 서비스 제품이다.** 운영 체계가 없으면 시니어에게 닿지 않는다.

### CS(고객 지원) 모델 — 시니어 특성 반영

| 단계 | CS 채널 | 응답 시간 | 인력 |
|------|--------|----------|------|
| **S1** | **전화 1개 회선** (주상윤·김경림 교대) + 앱 내 챗봇 | 평일 9~18시, 전화 30초 내 응답 | 0.5명 |
| **S2** | 전화 + 카카오톡 채널 + 앱 챗봇 → 전담 CS 2명 | 평일 9~21시, 3분 내 응답 | 2명 |
| **S3** | 24시간 긴급 전화 + 지자체 사회복지사 연계 | 응급 3분·일반 30분 | 5명 |

**시니어 특화 원칙**: "클릭 대신 전화" — 시니어의 50% 이상이 챗봇보다 전화를 선호한다는 전제로 **전화 창구를 S1부터 유지**.

### 사회복지사·보호자 협업 모델

- **S1**: 실증 지자체 사회복지사 2명과 주 1회 리뷰 미팅
- **S2**: 보호자용 월간 요약 리포트 이메일 발송 (부모님 활동 요약)
- **S3**: 지자체 사회복지사 대시보드 — 고위험군 스크리닝 자동 알림

### 장애 대응 체계

| 단계 | 기준 | 대응 |
|------|-----|------|
| **S1** | 99% 가용성 목표 | Sentry 알림 → 주상윤·BE 엔지니어 즉시 대응 |
| **S2** | 99.5% + 온콜 로테이션 | PagerDuty · 15분 내 대응 |
| **S3** | 99.9% + SLO 공식화 | 전담 SRE 1명 · 리전 페일오버 |

### 콘텐츠 운영 — "복지 소식 에디터"

- **S1**: 주 2회 혜택 카드 수동 작성 (주상윤·김경림 합 주 6시간)
- **S2**: 전담 에디터 1명 · 월 20건 발행
- **S3**: 지역별 맞춤 콘텐츠 + AI 초안 + 에디터 검수 (월 100건)

## 9. Go-To-Market 실행 플레이북

### 실증 지자체 1곳 섭외 — S1 성패의 핵심

**후보군** (주상윤 S0~S1 초기 6주 내 접촉):
1. 서울 내 초고령화 자치구 (종로·중구·강북)
2. 수도권 노인복지관 연계 (예: 성남시 노인종합복지관)
3. 디지털 복지 혁신 의지 강한 지자체 (공공데이터 활용 우수 지자체)

**접촉 경로**:
- 보건복지부·한국노인인력개발원 담당자 → 공식 소개
- AC 네트워크 통한 지자체 접점
- 시니어 관련 NGO(은빛복지회 등) 경유

**성사 조건**: S1은 **무상 파일럿 + 공동 홍보** 제공 대신, **사용자 50~200명 확보 · 담당자 협업**을 조건으로 제시.

**Plan B**: 지자체 확보 실패 시 → **지인 시니어 20명 + 가족 네트워크 30명 = 50명 클로즈드 베타**로 S1을 대체 시작. 지자체는 S2 시점에 재시도.

### 첫 100명 사용자 모집 전략 (S1)

| 채널 | 목표 | 방법 |
|------|------|------|
| 가족 네트워크 | 20명 | 팀원 부모·조부모 + 지인 5촌 이내 |
| 실증 지자체 | 50~100명 | 파일럿 참여자 모집 공고 + 사회복지사 추천 |
| 시니어 카페·경로당 | 30명 | 오프라인 홍보 (전단지·설명회) |
| 온라인 커뮤니티 | 20명 | 시니어 대상 블로그·카페 소개글 |

### B2G 확대 경로 (S2 → S3)

1. **S1 레퍼런스 확보** — 파일럿 지자체 6개월 운영 지표 (사용자 수·수급 전환율·만족도)
2. **S2 지자체 3곳 유료 계약** — 월 30~100만원 구독형. 세일즈 담당자 1명 채용.
3. **S2 말 한국노인인력개발원 MOU 제안** — 2026.4.16 원장 선언을 레버리지로.
4. **S3 중앙정부 PMO 참여** — AX-Sprint 예산(300억) 또는 후속 예산 공모 참여.
